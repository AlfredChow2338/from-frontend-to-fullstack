**Theme:** Node fundamentals

## Questions

1. Why commit `.env.example` but not `.env`? What could go wrong if `.env` is pushed to Git?
2. Name one way to read `process.env.MY_VAR` in a typed way.

---

## Answers

1. `.env` files may contain sensitive data such as API keys and secrets. `.env.example` is a `.env` template that helps collaborators to set up the projects locally but removes all the sensitive data. If the git project is public, attackers can steal the API keys or secrets that causes losses to the owners.

2. Declare global module or create a typed config module `config.ts` that map the values to .env file.

### Reviewer comments

- **Q1:** Correct and practical. Good explanation of the security risk: committing `.env` can leak credentials and lead to real abuse/cost. Nice distinction between a safe template (`.env.example`) and real secrets (`.env`).
- **Q2:** Acceptable direction. A typed config module is a solid approach. One concrete example to remember: define an interface/schema for required env vars, validate at startup, and expose typed getters so usage is `config.myVar` instead of raw string access everywhere.
