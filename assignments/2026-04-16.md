**Theme:** Node fundamentals

## Questions

1. What happens to an unhandled `Promise` rejection in Node? How does `try/catch` around `await` help?
2. In one sentence: what runs on the microtask queue vs the next tick?

---

## Answers

1. For node version < 16, there will be a console warning UnhandledPromiseRejectionWarning and the app keeps running.

For node version >= 16, unhandled rejection will log a fatal error and terminate the prcoess with non-zero exit code.

If the promise rejects, the event loop pushes a task to "resume" the function, but it marks the resumplation with an exception signal.

`try/catch` makes async code behave like traditional sync code, which the next line of the unhandled code will not execute until the await promise settles.

The main thread will not block. The function state will be saved to heap and move to other tasks in the event loop.

2. `process.nextTick` execute immediately after after the current operation completes before the microtask queue is processed.

- **Q1:** Good core understanding: unhandled promise rejections are dangerous and can terminate the process depending on Node/runtime settings. `try/catch` around `await` is exactly the right pattern to keep errors explicit and prevent unexpected crash paths.
- **Q2:** Acceptable direction. You correctly pointed out `process.nextTick` runs before normal promise microtasks. To complete the contrast: Promise callbacks (`then/catch/finally`, `queueMicrotask`) run in the microtask queue; timers/I/O callbacks run later in event-loop phases.
