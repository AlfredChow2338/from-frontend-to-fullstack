## Questions

1. What is the difference between compiling TS with `tsc` and running with `ts-node` / `tsx`?
2. What does `"strict": true` in `tsconfig.json` change for you as a developer?

## Answers

1. `tsc` is TypeScript CLI which transpiles the file based on tsconfig.json for distribution. You still need to run `node ${file}` to run the script.

`ts-node` starts from the entry file and transpile the file through the tree-based import export for development.

`tsx` is similar to `ts-node` but it will not do the type check without tsconfig.json. It also has fast compilation speed by using esbuild.

2. "strict": true enable strict type checking in tsconfig eg. no implicit any, strict null checking, apply "use strict" in js files etc.

It serves as a safety net and force typescript good practice to make the codes more type safe and easier to maintain.

### Reviewer comments

- **Q1:** Good distinction. `tsc` compiles TypeScript into JavaScript output (often for build/distribution), while `ts-node` / `tsx` execute TS directly in dev workflows. Nice note that `tsx` is fast because it uses esbuild; one nuance: type-checking is typically done separately with `tsc --noEmit` in many teams.
- **Q2:** Correct core idea: `"strict": true` enables a stricter set of type checks and catches more bugs earlier. Minor correction: `"use strict"` in emitted JS is controlled by compilation/module output behavior, not directly by the `strict` type-checking flag itself.
