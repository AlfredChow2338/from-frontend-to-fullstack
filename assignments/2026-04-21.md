## Questions

1. Give one example each for 201, 204, 409 in a REST API.
2. What does `422 Unprocessable Entity` imply about the request body?

---

## Answers

1. In the context of Note resouces.

201 Created is when you created a new note successfully like POST /note.

204 No content is also success but there is no content return for example DELETE /note/:noteId.

409 Conflicts means when you create a new note but there is already an existing note having conflicts.

2. 422 implies the request body is valid and well-form but the server cannot process it due to some business logic semantic errors.

### Reviewer comments

- **Q1:** Good examples and mostly correct mapping: `201` on successful create, `204` on successful delete/update with no response body, and `409` for state conflicts (e.g., unique key/title already exists). Nice practical context with notes. Small polish: `resource` spelling and prefer collection path `POST /notes`.
- **Q2:** Correct core meaning: payload is syntactically valid but semantically unacceptable for current business rules. Good distinction from `400` (malformed/invalid format-level validation). In many Nest setups, validation defaults to `400` unless explicitly mapped to `422`.
