## Questions

1. Which HTTP methods are idempotent? Why does that matter for retries?
2. When is PUT _not_ a good fit compared to PATCH?

---

## Answers

1. GET PUT DELETE HEAD OPTIONS TRACE are idemoptent because the effect of making single request and making multiple requests is the same, which is retry-safe.

POST and PATCH are not idemoptent instead as the outcome may vary.

2. PUT is replacing the targeting resouce and overriding the existing one.

PATCH is partially updating some attributes of the resouces.

### Reviewer comments

- **Q1:** Correct list and core reasoning: idempotent methods produce the same server state when repeated, which is why retries are safer for network failures/timeouts. Nice mention of retry safety. Minor typo polish: `idempotent`.
- **Q2:** Your PUT vs PATCH contrast is right. To answer the question more directly: PUT is **not** a good fit when the client has only partial fields (you might unintentionally overwrite omitted fields) or when full replacement semantics are not intended. PATCH is better for sparse updates.
