## Questions

1. When do you use POST vs PATCH for a “Note” resource?
2. Which HTTP status would you return for validation errors vs “not found”?

---

## Answers

1. POST is create a new resource. POST /note is to create a new note.

PATCH is to update an existing resouce. PATCH /note/:noteId is to update the note with the corresponding note id.

Additionally, PUT is also update an existing resouce. PUT /note/:noteid is to replace the existing note instead of partially update some information (PATCH).

2. Validation errors will return 400 Bad Request and not found will return 404 Not Found.

### Reviewer comments

- **Q1:** Correct mental model: **POST** creates (typically `POST /notes` on the **collection**; plural is conventional), **PATCH** applies a partial update on an existing id, **PUT** replaces the whole representation. You nailed PUT vs PATCH. Tiny polish: spell **resource**; optional REST nuance — avoid `POST /note/:id` for “create”; creation is usually collection POST with body, server assigns id → **201** + `Location` when you get there.
- **Q2:** Right pair: **400** for bad/invalid client input (validation), **404** when the target resource id does not exist. Optional: some APIs use **422** for “syntax OK but semantically invalid”; Nest often still maps validation to **400** via `ValidationPipe` unless you customize. See [MDN HTTP status](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status).
