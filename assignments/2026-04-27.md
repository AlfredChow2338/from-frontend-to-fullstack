## Questions

1. What is the responsibility of a Nest module vs a controller vs a service?
2. What is a DTO and why validate at the boundary?

---

## Answers

1. Module (Department) organises and encapsulates the features (services, controllers, entities).

Controller (Waiters) handles HTTP requests and responses.

Servies (Chef) is the core business logic implementations.

2. DTO refers to data transfer object, which is the definition of the shapes of the data that comes in and out.

DTOs in Nest are often classes because of preserving entities in compiled javascript, which can be used for runtime validations.

### Reviewer comments

- **Q1:** Clear split: the module bundles and wires the feature (controllers, providers, imports), the controller is the HTTP edge, the service holds business logic — your Department / Waiter / Chef framing fits. Fix the typo **Servies → Services**. Optional nuance: the module is mainly where Nest registers **dependency injection** and composes other modules; it should stay thin compared to services.
- **Q2:** Good core: DTO as the shape of data in/out, and **classes** so `class-validator` + `ValidationPipe` still work after TypeScript is erased. To tie to “boundary”: HTTP bodies are **untrusted**; validating at the controller/pipe layer keeps services clean and yields **consistent 400** responses before domain code runs. See [NestJS validation](https://docs.nestjs.com/techniques/validation) and [Pipes](https://docs.nestjs.com/pipes).
